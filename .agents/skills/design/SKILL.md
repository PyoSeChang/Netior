# Design-First Feature Development

## Golden Rule

**코드를 한 줄도 쓰지 않는다** — 설계 문서가 사용자에게 승인받기 전까지.

## Process

### Phase 1: 구체적 사례 수집

기능을 추상적으로 정의하지 않는다. 실제 사용자가 보는 화면을 먼저 정의한다.

1. 이 기능의 **일반적인 형태**가 이미 존재하는가? (간트 차트, 캘린더, 칸반 등)
   - 존재하면 **그 일반적 형태를 먼저 조사**한다. 직접 써보거나, 스크린샷을 찾거나, 기존 코드(Culturium 등)를 읽는다.
   - "일반적"이란 뜻을 정확히 이해한다. 사용자가 "일반적인 타임라인"이라 하면 그건 구글 캘린더/간트 차트 수준의 완성도를 기대하는 것이다.
2. 이 프로젝트에서의 **구체적 use case 3개 이상**을 나열한다.
3. 각 use case에서 사용자가 **화면에서 보는 것**을 글로 묘사한다 (ASCII 와이어프레임 또는 자연어).

### Phase 2: 시각 설계

코드가 아닌 **자연어로** 설명한다. TypeScript 인터페이스를 쓰지 않는다.

1. **화면 구조** — 고정 영역, 스크롤 영역, 레이어 순서
2. **구성 요소** — 각 시각적 요소가 뭘 보여주는지, 어디에 위치하는지
3. **크기/비율** — 줌이나 화면 크기에 따라 뭐가 고정이고 뭐가 변하는지
4. **데이터 표시** — 실제 데이터 예시로 라벨이 어떻게 보이는지 ("2024년 3월", "15일" — "3년", "4년" 이런 추상적 라벨 금지)

### Phase 3: 인터랙션 정의

**모든 사용자 입력과 그에 대한 시스템 반응을 빠짐없이 나열한다.**

표 형태:

| 입력 | 동작 | 비고 |
|---|---|---|
| 마우스 휠 | ? | |
| Ctrl+휠 | ? | |
| Shift+휠 | ? | |
| 빈 영역 클릭 | ? | |
| 빈 영역 드래그 | ? | |
| 노드 클릭 | ? | |
| 노드 더블클릭 | ? | |
| 노드 드래그 (X) | ? | |
| 노드 드래그 (Y) | ? | |
| 노드 우클릭 | ? | |
| 우클릭 → 생성 | ? | |
| Delete 키 | ? | |
| Ctrl+A | ? | |
| 줌 인 | ? | |
| 줌 아웃 | ? | |

빠진 인터랙션이 있으면 사용자에게 물어본다.

### Phase 4: 데이터 흐름

1. 이 기능에 필요한 **데이터가 어디서 오는지** (DB, 사용자 입력, 계산)
2. 데이터가 **어떤 변환을 거치는지** (date string → epoch days 등)
3. 사용자 조작 결과가 **어디에 저장되는지** (어떤 테이블, 어떤 필드)
4. **기존 데이터 모델로 충분한지**, 부족하면 뭘 추가해야 하는지

### Phase 5: 제약 조건과 Edge Case

1. 데이터가 없을 때 (빈 상태)
2. 데이터가 극단적일 때 (0개, 1000개, 음수, 역순)
3. 레이아웃 전환 시
4. 관련 설정이 안 됐을 때 (매핑 누락 등)

### Phase 6: 설계 문서 작성

위 Phase 1-5의 결과를 **별도 파일**로 작성 (`DESIGN-{feature}.md`).

문서에 포함:
- 시각 구조 (Phase 2)
- 전체 인터랙션 표 (Phase 3)
- 데이터 흐름 (Phase 4)
- Edge case (Phase 5)

**사용자 승인을 받는다.** 승인 없이 구현 시작 금지.

### Phase 7: 구현 계획 (Plan Mode)

설계 승인 후에야 Plan Mode로 진입.
- 파일별 변경 내용
- Phase별 검증 항목 (촘촘하게)
- 구현 순서

## Anti-patterns

이 스킬이 방지해야 하는 실수:

- **추상적 설계** — "숫자 직선에 값을 배치" 대신 "2024년 3월 15일이 화면 어디에 어떻게 보이는지" 수준으로 구체적이어야
- **도메인 독립성 집착** — 먼저 가장 일반적인 형태(달력, 간트 차트 등)로 동작하게 만들고, 그 다음에 설정 가능하게 확장
- **코드로 설계** — TypeScript 인터페이스나 코드 블록으로 설계하면 구현 디테일에 빠져서 큰 그림을 놓침. 자연어로 설계
- **인터랙션 누락** — "드래그하면 이동" 한 줄로 퉁치지 말고, X축/Y축 분리, 줌 영향, 저장 시점, 시각적 피드백까지 정의
- **기존 형태 무시** — "일반적인 타임라인 만들어줘"라고 하면 실제 간트 차트/캘린더 앱을 참고해야지, 백지에서 발명하면 안 됨
